Technical Specification: Personal Workout Companion

1. Purpose

This document defines the implementation-level specification for Personal Workout Companion, a single-user, self-hosted, mobile-first workout logging application.

The audience is a developer implementing the system. This specification defines:

- System architecture
- Data model
- Client behavior
- Server behavior
- External sync behavior
- API contracts
- Acceptance criteria

2. Scope

The system must support:

- Manual Strength workouts
- Manual Cross Training workouts
- Imported Cardio workouts from Garmin
- Big 3 workout assistance for Squat, Bench Press, and Deadlift
- Plate inventory and plate-loading calculations
- Offline workout drafting and later synchronization
- Import and reconciliation of external Garmin activities
- Basic analytics for strength and cardio history

Out of scope for the first implementation:

- Multi-user support
- Social features
- Program builder or workout templates beyond Big 3 defaults
- Native mobile apps
- Integration with more than one external provider
- Official Garmin developer-program integration

3. Fixed Technical Decisions

- Backend framework: Flask
- Database: SQLite
- Rendering model: server-rendered Jinja templates with targeted JavaScript
- Client offline storage: IndexedDB
- PWA support: service worker and manifest
- Background processing: scheduled Python sync job or sidecar process
- Authentication model: single local application user
- Garmin adapter: unofficial `garminconnect` Python library

These decisions are not open within this specification.

4. System Architecture

4.1 Runtime Components

The implementation consists of five runtime components:

- Web server: Flask app serving HTML pages and JSON endpoints.
- SQLite database: persistent store for workouts, config, external activities, and sync state.
- Client application shell: HTML, CSS, and JavaScript running in the browser.
- Client offline store: IndexedDB for draft state and queued write operations.
- Sync worker: scheduled Python process that uses `garminconnect` to pull Garmin activities and reconcile them against local data.

4.2 Component Responsibilities

Web server:

- Serve dashboard, workout, summary, admin, and analytics pages.
- Expose JSON endpoints for draft creation, set operations, finalization, configuration, and sync actions.
- Enforce validation and idempotency for client operations.
- Compute exercise suggestions, plate-loading responses if needed server-side, and analytics queries.

SQLite database:

- Persist canonical application state.
- Persist audit timestamps for reconciliation actions.
- Persist sync checkpoints and external activity records.
- Persist idempotency markers for client operations.

Client application shell:

- Render mobile-first workflows.
- Maintain active workout state from local storage.
- Queue user actions while offline.
- Replay queued actions when connectivity is restored.
- Preserve timer state across reloads.

Sync worker:

- Authenticate to Garmin through the `garminconnect` library token flow.
- Fetch activity data using a checkpoint-based window.
- Upsert external activities.
- Reconcile activities against workouts.
- Create or update pending imports.

4.3 Concurrency Model

The application is single-user but may have concurrent writers:

- Browser requests
- Background sync process

Implementation constraints:

- Keep write transactions short.
- Use SQLite WAL mode.
- Retry transient database locked errors with bounded backoff.
- Avoid long-running transactions during sync reconciliation.

5. Domain Model

5.1 Enumerations

Workout types:

- `strength`
- `cross_training`
- `run`

Workout statuses:

- `draft`
- `finalized`
- `archived`

Set types:

- `normal`
- `amrap`
- `for_time`

External activity statuses:

- `discovered`
- `pending_review`
- `linked`
- `dismissed`

External providers:

- `garmin`

5.2 Core Entities

Workout:

- Represents a manual or imported workout session.
- Has a stable UUID generated by the client for manually created workouts.
- May link to at most one external activity, and an external activity may link to at most one workout.

Workout set:

- Represents a single logged set within a workout.
- Has a stable UUID generated by the client before server sync.

External activity:

- Represents a Garmin activity mirrored locally.
- Is unique by provider and provider activity ID.

Pending import:

- Is not a separate table.
- It is an `external_activities` row whose status is `pending_review` and which is not linked to a workout.

Client operation:

- Represents an idempotent action emitted by the client and applied by the server.

6.3 Database Rules

- All timestamps must be stored in UTC ISO 8601 format.
- SQLite foreign keys must be enabled.
- `updated_at` must be refreshed on every mutation.
- A finalized workout may be updated only for:
  - External metric attachment
  - Notes correction if an explicit future edit feature is added

7. Client-Side Technical Behavior

7.1 IndexedDB Stores

The browser must maintain at least these logical stores:

- `active_workout_drafts`
- `client_operation_queue`
- `timer_state`
- `cached_config`

7.2 Active Draft Model

An active draft in IndexedDB must contain:

- `workout_id`
- `workout_type`
- `status`
- `started_at`
- `exercise_rows`
- `set_rows`
- `pending_operation_ids`
- `timer_state`
- `last_local_write_at`

7.3 Draft Lifecycle

On workout start:

- Generate a client workout UUID.
- Create local draft immediately.
- Render the active workout UI from local state.
- Attempt to synchronize draft creation in the background.

On every edit:

- Apply mutation to IndexedDB first.
- Append an operation to `client_operation_queue`.
- Attempt async flush if online.

On reload:

- Restore the draft entirely from IndexedDB.
- Resume timer calculations based on stored start timestamps, not in-memory counters.

On finalize:

- Prevent duplicate submissions using a local `finalize_in_flight` guard.
- Queue the finalize operation if offline.
- Mark local draft as `finalized-pending-sync` until acknowledged.

`finalized-pending-sync` is a client-local transient state only. It must never be persisted to the server as a `workouts.status` value.

7.4 Timers

Timer state must be persisted as timestamps, not ticking counters.

Required timer fields:

- `mode`
- `started_at`
- `stopped_at`
- `target_duration_seconds` for rest timers

This allows exact restoration after refresh and prevents drift from suspended tabs.

8. Server-Side Behavior

8.1 Validation

The server must reject:

- Unknown workout types
- Unknown set types
- Invalid UUID formats
- Negative reps
- Negative durations
- Negative plate counts
- Finalize requests without a feeling score
- Set updates against nonexistent workouts

The server may accept null `weight_kg` for time-only activities if consistent with the set type.

8.2 Idempotency

Every write sent through the operation queue must include:

- `operation_id`
- `workout_id`
- `operation_type`
- `client_timestamp`
- `payload`

Server rules:

- If `operation_id` already exists with status `applied`, return success with prior result.
- If `operation_id` already exists with status `rejected`, return the same rejection.
- Otherwise, insert the operation log row and process exactly once.

Write model:

- The operation queue is the canonical client write path for workout draft creation, set mutation, and workout finalization.
- `POST /api/client-operations` must support the same write semantics whether the client is currently online or replaying offline work.
- Resource-specific write endpoints may exist as thin server adapters for the same operations, but they must require the same idempotency fields and produce equivalent effects.
- The implementation must not define one set of semantics for queued writes and a different set for direct online writes.

8.3 Workout Mutability Rules

Draft workouts:

- Can receive create, update, delete set operations.
- Can be finalized.

Finalized workouts:

- Reject set mutations.
- Allow external metric attachment from reconciliation.

Archived workouts:

- Read-only.

9. External Garmin Integration via `garminconnect`

The first implementation uses the unofficial `garminconnect` Python package as the Garmin adapter. This is an implementation dependency, not a stable vendor contract, so the sync subsystem must tolerate upstream Garmin API and authentication changes.

9.1 Authentication and Token Storage

Preferred login model:

- Use the `garminconnect` library with token-based authentication and persisted token storage.
- Treat raw Garmin username and password as bootstrap-only secrets used to establish or refresh tokens if required by the library.
- Do not persist raw Garmin credentials in SQLite.
- Do not log credentials, MFA codes, tokens, or decrypted secret material.

If local token persistence is needed:

- Store tokens in a filesystem location with restrictive permissions, or encrypt them before storing in SQLite.
- If SQLite is used for token storage, store encryption key material outside SQLite, for example in an environment variable or local secret file outside the database.
- The implementation must support token invalidation and re-authentication without corrupting sync state.

9.2 Adapter Reliability Rules

Because `garminconnect` is unofficial:

- Encapsulate all Garmin-specific calls behind a dedicated adapter module.
- Do not spread `garminconnect` response shapes throughout the application.
- Normalize Garmin activity payloads into the local `external_activities` schema before reconciliation.
- Persist the raw normalized payload or raw upstream payload for debugging.
- Fail the sync run cleanly if Garmin authentication or payload parsing breaks.
- Record the failure in `sync_checkpoints.last_status` and `last_error`.

9.3 Sync Trigger Rules

Two sync modes must exist:

- Automatic scheduled sync every 15 to 30 minutes.
- User-initiated or dashboard-triggered sync subject to cooldown.

Dashboard-triggered sync cooldown:

- Default minimum interval: 10 minutes since `last_attempted_sync_at`.

The first implementation must use polling. It does not depend on Garmin webhooks or push delivery.

9.4 Fetch Strategy

Sync must not use "last N activities".

Algorithm:

1. Read `last_successful_sync_at` for Garmin.
2. Compute `window_start` as `last_successful_sync_at - 24 hours`, or `now - 14 days` if no checkpoint exists.
3. Fetch activities from Garmin for `window_start` to `now` using the `garminconnect` adapter.
4. For each fetched activity, map the Garmin payload into the local normalized external-activity shape.
5. Upsert each activity into `external_activities` by `(provider, provider_activity_id)`.
6. Reconcile each changed or new activity.
7. Update checkpoint only after a successful pass.

Fetch rules:

- The adapter must prefer stable Garmin activity identifiers over timestamps for deduplication.
- The overlap window is intentional and must be retained even after successful syncs.
- Re-fetching previously seen activities must be safe and idempotent.
- If the adapter can access detailed activity payloads only via per-activity follow-up calls, those calls must be bounded to newly seen or changed activities where possible.

9.5 Reconciliation Rules

For each external activity:

1. If an existing `external_activities` row is already linked to a workout, refresh metrics on the linked workout and stop.
2. Find candidate workouts where:
   - `started_at` is within 15 minutes of the external activity start time.
   - Workout type is compatible with activity type.
3. If exactly one candidate exists, auto-link.
4. If more than one candidate exists, set status to `pending_review`.
5. If no candidate exists, set status to `pending_review`.

Compatibility rules:

- Garmin run, trail run, treadmill run -> `run` or existing `cross_training`
- Garmin cycling -> `cross_training`
- Garmin strength -> existing `strength` only

The first implementation may keep compatibility as a Python mapping table.

9.6 Dismissal Rules

When a pending import is dismissed:

- Set status to `dismissed`
- Preserve the row
- Record `dismissed_at`

Dismissed activities must not be re-created on later sync runs because uniqueness is based on provider activity ID.

9.7 Accept and Link Rules

Pending import actions must behave as follows:

- `accept` creates a canonical `workouts` row when no linked workout already exists.
- `link` attaches the external activity to an existing compatible workout.
- Both actions are idempotent for the same target result.

`accept` behavior:

1. Require the `external_activities` row to be in `pending_review` unless the operation is a no-op repeat of a previously accepted import.
2. If `linked_workout_id` is already populated, return the existing linked workout and do not create another.
3. Otherwise create a new `workouts` row with:
   - `id` as a server-generated UUID
   - `type` as `run`, unless `type_override` is provided and allowed by the compatibility rules
   - `status` as `finalized`
   - `started_at` copied from the external activity
   - `ended_at` copied from the external activity when available
   - `source` as `external_import`
   - `notes` and `feeling_score` as null by default
4. Set `external_activities.linked_workout_id` to the new workout ID.
5. Set `external_activities.status` to `linked`.
6. Do not create any `workout_sets` rows for an accepted run workout in the first implementation.

`link` behavior:

1. Require `workout_id`.
2. Reject linking if the target workout already has a different external activity linked.
3. Reject linking if the target workout type is incompatible with the external activity type under the compatibility rules.
4. On success, set `linked_workout_id` and `status = linked`.

`type_override` rules:

- `type_override` may be used only on `accept`.
- Allowed values are `run` and `cross_training`.
- `type_override` must still satisfy the compatibility rules for the Garmin activity type.
- `type_override = strength` is not allowed for `accept`; Garmin strength may only link to an existing manual `strength` workout.

9.8 Failure and Recovery Behavior

Sync runs must distinguish at least these failure classes:

- Authentication failure
- Network failure
- Garmin upstream schema or parsing failure
- Local database failure

Required behavior:

- `last_attempted_sync_at` must be updated on every attempted run.
- `last_successful_sync_at` must be updated only after a successful end-to-end fetch and reconciliation pass.
- `last_status` must record `success` or a stable failure category.
- `last_error` must store a short diagnostic message safe for local display.
- A failed sync run must not partially advance the checkpoint past unfetched data.
- Re-running sync after failure must be safe without manual cleanup.

10. Suggestion and Calculation Algorithms

10.1 Big 3 Prefill

For Squat, Bench Press, and Deadlift:

- Query the most recent finalized workout set for the selected lift.
- Suggest:
  - `weight_kg = last_weight_kg + configured_increment`
  - `reps = last_reps`

If no history exists:

- Leave values blank or use developer-defined defaults from config.

10.2 Exercise Suggestions

Inputs:

- `previous_exercise_name`
- `query`

Ranking inputs:

- Transition frequency in finalized workouts over trailing 90 days
- Global exercise frequency over trailing 180 days
- Prefix match against the current query

Required behavior:

- Prefix matches must be included even if historical score is low.
- New names must not enter the dictionary until the workout is finalized.

10.3 Plate Loading

Inputs:

- Target weight
- Barbell weight
- Plate inventory

Rules:

- Compute plates per side.
- Never allocate more plates of a given weight than exist in inventory.
- Prefer exact match.
- If multiple exact solutions exist, prefer the solution with fewer total plates.
- If no exact solution exists, return nearest lower and nearest higher achievable targets within a configurable search radius.

The algorithm may be implemented on either client or server, but it must be deterministic.

13. Acceptance Criteria

13.1 Workout Logging

- Starting a workout creates a local draft even if the network is offline.
- Reloading the page restores the draft exactly.
- A set created offline syncs successfully later without duplication.
- Finalizing a workout prevents further set edits through normal endpoints.

13.2 Garmin Sync

- A Garmin activity fetched twice does not create duplicate external activity rows.
- A clear one-to-one match auto-links successfully.
- Ambiguous matches remain pending for review and are not silently linked.
- Dismissed imports do not reappear on the next sync.

13.3 Suggestions and Plate Loading

- Big 3 prefill uses the latest finalized history for the chosen lift.
- Exercise suggestions include prefix matches and history-based ranking.
- Plate loading never exceeds configured inventory counts.
- Plate loading returns exact or nearest achievable weights deterministically.

13.4 Operational Reliability

- The app remains usable when offline during an active workout.
- Client operation replay is idempotent.
- SQLite locking during sync does not corrupt workout data.

14. Implementation Notes

- Use UTC everywhere internally. Convert to local time only in rendering.
- Prefer explicit status fields over inferred state from nullable columns.
- Keep Garmin raw payloads for debugging and future parser changes.
- Do not merge external activity metrics directly into workout-set rows.
- Keep the first implementation simple. Avoid introducing message queues, Redis, or a SPA framework.

15. ADR-001: Technology Stack

Status:

- Accepted

Decision:

- Implement the system with Flask, SQLite, Jinja, targeted JavaScript, IndexedDB-backed offline support, and a scheduled Python Garmin sync worker.

Rationale:

- This stack satisfies the local-first and self-hosted constraints with minimal operational overhead.

Consequences:

- Offline behavior must be implemented deliberately on the client.
- SQLite write contention must be managed carefully.
- The application stays simple to deploy and inspect.
